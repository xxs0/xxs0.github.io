理想的HTTP性能 -- 笔记

发表于 2016-07-10   |   分类于 Web前端

最近看了一篇关于理想HTTP性能的的文章，讲解很详细，看完做个笔记。(文中图片尚未添加)

  • 原文链接：Ideal HTTP Performance
  • 作者：Mark Nottingham.IETF HTTP Working Group现任主席，W3C TAG成员.

起因

对于web性能隐喻目标是减少终端用户（end-user）可察觉的延迟，在用户和交互前面尽快获得页面。

所以HTTP理想的协议交互应该是这样的。

页面加载应该发送最少的数据给服务器(minimal amount of data to the server)，并且在最可能少的往返(least possible number of round trips)中从服务器获取可能需要的最少数据(minimal amount of data possible needed from it)。

任何一个方向上额外的数据意味着要更多的时间去传输和更多机会遇到如拥塞和丢包，从而严重影响性能。

额外的往返由于协议“繁琐”（chattiness）增加延迟，特别是移动网络状况下（100ms是你能期待的做好结果；详见Ilya的文章）

如果这是理想的状态，HTTP可采用什么措施？我们该怎么提高？

HTTP/1.1

可以有一系列的原因说HTTP/1.1是一个好的协议，但是不幸的是现代Web工作的方式意味着性能不是其中的一个。一个传统页面加载如图

HTTP/1是‘闲聊式的’（chatty），客户端需要多次返回服务器发现新内容。首先是HTML，然后是CSS和JavaScript，每个增加一次或多次往返的延迟
不符合我们“least number of round trips ”的理念

更远的说；发送请求增加了许多数据，不符合“minimal amount of data to the server”的理念。因为如Referer, User-Agent，course Cookie这些冗长的头部在每次请求都重复，

最后，由于HTTP/1的队头阻塞（Head-of-line_blocking），一个常用的实践是把多个资源放入一个CSS雪碧图，inline和concatenation，这是一个很漂亮的HTTP/1性能hack，但是这也有许多代价，它们下载了许多不需要的数据。

归结起来，HTTP/1.1不全是坏的，性能聪明。例如，有缓存(caching),允许你避免在有一个fresh copy时一直是用网络。conditional requests允许你在缓存中有stable copy时避免传输大块数据。

HTTP/2

HTTP/2 尝试用一些方法解决HTTP/1.1中的问题

  • Full multiplexing 意味着队头阻塞不再是问题，你能在单个HTTP连接上加载WEb页面，不用担心产生很多请求。数据浪费‘优化’技术（Data－wasting optimisation techniques）可以抛到脑后。

  • Header compression 移除了冗长头部引起的per-message overhead，现在能放置数十甚至数百请求在一对IP包中，在两端都更接近“minimal data”的理念。

  • HTTP/2 Server Push 允许服务器预测客户端需要的内容，避免聊天式往返。

所以HTTP/2交互应该如下图

图中能看到服务器发送CSS，JavaScript，images给客户端不用询问。它知道客户端可能要询问，所以是用Server Push发送一个合成（synthetic）请求／响应对给客户端，减少一个往返。它是一个less “chatty” protocol，更充分利用网络。

HTTP/2还是有许多待解决问题。特别是围绕何时推送（when to push something）。

HTTP/2 + Cache Digests

关于Server Push的一个普遍问题是“what if the client already has a copy in cache?”， 因为推送是内在的推测（inherently speculative），总会遇到你正在发送的东西不是浏览器需要的。

HTTP/2 允许客户端在这种情况下用一个RESET_STREAM取消推送，即使这样，仍然有大致一个往返的垃圾数据在flight中原本用于更好的事情。记住，理念是只发送客户端需要显示在页面中的数据。

一个提出的解决方案是对客户端是用一个compact Cache Digest 去告诉服务器它在缓存中已经有什么，以致服务器知道需要什么。

pic

因为Cache Digests是用Golumb Compressed Sets,它是切实可行的认为，浏览器缓存能告诉服务器少于1000字节的内容关于它已经有了什么，发送它给服务器在一个连接的开头的一些包中。

现在，我们已经避免繁琐的额外round trips，the wasted data of concatenation, inlining，累似的hacks和推送不需要的请求的垃圾数据.

Cache Digests只是一个提议，但是在HTTP社区对它们有大量公正的兴趣，希望在不远的将来能看到它们。

TCP

到现在为止，我们还没讨论浏览器用来加载Web页面的其他协议的性能影响。
TCP在HTTP开始前需要三次握手。

pic
这意味着最小的一个全往返花费在建立连接，给每个新连接增加了繁琐。

TCP Fast Open允许应用程序在SYN和SYN＋ACK包上发送数据去避免那种情况。不过，它现在只被Linux和OSX支持，而且，是用TFO with HTTP有一些棘手的后果，社区正在开始研究。

即，TFO不能保证通过SYN包发送的数据只显示一次，它是有漏洞的重复（由于再次发送）甚至恶意重放攻击（replay attacks），所以，HTTP POST对于TFO连接上的第一个请求不是一个好主意。更有问题的是，一些GETs也有副作用，但是浏览器没有好的方式检测是哪个URLs做的。

TLS

TLS增加了另一种连接启动式聊天，在TCP握手完成后。
它看上去是这样的
pic

在HTTP能发送数据前有两个完整往返。session tickets能让你避免一个往返。
pic

不久，TLS1.3将允许当客户端到服务器之前一个“零往返”（zero round trip）握手。换而言之，HTTP将能够在第一个往返发送数据，避免任何增加的延迟。然而就像TCP Fast Open， 你需要能确定在第一个往返中重复的数据不会坏事。

HTTP/next

TCP Fast Open 和 TLS1.3都是用来减少在和服务器建立新连接的聊天（chattiness）的方式。另一种方法是尽量重复利用已经打开的连接。

最后，怎么利用HTTP/2’s connection coalescing的讨论更激烈；不仅是它帮助避免打开新连接的开销，也让存在的连接更高效，TCP在长寿命、忙流动下工作最佳。

这包括事情像pushing certificates to the client, 证明连接能被它原来协商的更origins。

一个更激进的变革正在被讨论，把TCP换成UDP，如QUIC。有很多移到QUIC上，但是从性能的角度，能偶0往返握手不需要更改客户端操作系统是很有吸引力的。而且，能够在缓存中无序访问数据意味着TCP中的队头阻塞（它是个有序协议）不再是问题；你能在丢包后摘取HTTP消息（或者片段）更快速的发送给用户。

QUIC还在早期，所以我们不能一段时间或一直看不到一个以UDP为基础（UDP－based）HTTP规范和广泛部署。讨论的一个可能的未来是QUIC是一个我们用来学会更多我们想要的TCP性能的测试平台，我们能把它们用到Web。那个在一个异构互联网中怎么可能仍需观察。

CSS居中-完整指南

发表于 2016-06-16   |   分类于 Web前端

来源链接：Centering in CSS: A Complete Guide

水平方向

行内元素

.center-children {
text-align: center;
}

  • 对于inline, inline-block, inline-table, inline-flex等有效

块级元素

.center-me {
margin: 0 auto;
}

  • 无论你想要居中的块级元素或父元素的宽度是多少，都有效。
  • 不能float一个元素到中间 trick though

多个块级元素

  • 如果有两个或多个块级元素需要在一列水平居中，用一个不同的display类型。

    .inline-block-center div {
    display: inline-block;
    text-align: left;
    }
    .flex-center {
    display: flex;
    justify-content: center;
    }

垂直方向

CSS中垂直方向居中有一点点困难

单行

  • 有时inline/text 元素能垂直居中，只因为上下padding1相等。
  • 使用line－height 等于高度

    .center-text-trick {
    height: 100px;
    line-height: 100px;
    white-space: nowrap;
    ｝

多行

  • 相等的padding－top和padding－bottom也能给多行文本居中效果。

  • 如果无效，可以是用table cell

.center-table {
display: table;
height: 250px;
background: white;
width: 240px;
margin: 20px;
}
.center-table p {
display: table-cell;
margin: 0;
background: black;
color: white;
padding: 20px;
border: 10px solid white;
vertical-align: middle;
}

  • 如果table－like无效，可以是用flexbox，一个单flex-child能在flex－parent中轻易居中

.flex-center-vertically {
display: flex;
justify-content: center;
flex-direction: column;
height: 400px;
}

记住，只有如果父容器有一个fixed height（px, %,etc）才有意义

  • 如果以上都无效，可以使用“ghost element”技术，

.ghost-center {
position: relative;
}
.ghost-center::before {
content: “ “;
display: inline-block;
height: 100%;
width: 1%;
vertical-align: middle;
}
.ghost-center p {
display: inline-block;
vertical-align: middle;
}

块级元素

知道元素高度

.parent {
position: relative;
}
.child {
position: absolute;
top: 50%;
height: 100px;
margin-top: -50px; // account for padding and border if not using box-sizing: border-box;
}

元素未知高度

.parent {
position: relative;
}
.child {
position: absolute;
top: 50%;
transform: translateY(-50%);
}

使用flexbox

.parent {
display: flex;
flex-direction: column;
justify-content: center;
}

水平、垂直居中

元素 fixed width 和height

  • 在绝对定位在50%／50%处后设置负margin 等于宽或高的一半。它有很好的跨浏览器支持

.parent {
position: relative;
}
.child {
width: 300px;
height: 100px;
padding: 20px;
position: absolute;
top: 50%;
left: 50%;
margin: -70px 0 0 -170px;
}

元素有未知宽、高

  • 未知宽高，可是用transform属性和一个50%的负translate。

.parent {
position: relative;
}
.child {
position: absolute;
top: 50%;
left: 50%;
transform: translate(-50%, -50%);
}

使用flexbox

.parent {
display: flex;
justify-content: center;
align-items: center;
}

addEventListener vs onclick -- 笔记

发表于 2016-05-08   |   分类于 Web前端

Event Listeners (addEventListener or IE’s attachEvent)

早期IE版本（小于9），使用attachEvent方法

ele.attachEvent('onclick', function() { /* do something * / });

大多数浏览器(包含IE9及以上)，可以使用addEventListener
ele.addEventListener(‘click’, function() { / do something / }, false);

使用DOM Level2事件，理论上能在任意单个元素上获取无限个事件。唯一的实际限制是客户端内存和其他性能考虑。每个浏览器都不一样
上面的例子可以用匿名函数展示。也可以用函数引用或者闭包监听事件。

var myFunctionReference = function() { /* do something * / };
ele.attachEvent('onclick', myFunctionReference);
ele.addEventListener('click', myFunctionReference, false);

addEventListener另一个重要特性是最后的参数，它控制对冒泡事件的反应。false对95%的情况有效。attachEvent没有相同的参数，或者当使用行内事件。

inline event

所有支持js的浏览器中，你都能在行内放置事件监听

<a id="testing" href="#" onclick="alert('do something inline');">Click me</a>

大多数有经验的开发者鄙视这种方法，但是它能把事做成，而且简单直接。这里不用闭包或匿名函数，你控制的范围被限制。

另一个方法等同行内js，除了你有更多控制范围，也能使用匿名函数，函数引用和闭包。

ele.onclick = function() { /* do something * / };

行内事件的重要缺点是不如事件监听，你只能操作分配的一个行内事件。行内时间被存储喂一个键值对，意味着它能被重写。

var ele = document.getElementById('testing');
ele.onclick = function () { alert('do something #1'); };
ele.onclick = function () { alert('do something #2'); };

当你点击元素，只能看到“do something ＃2”，重写了第一个onclick属性，也重写了原来的行内HTML属性。检查这里

那个最佳

这个问题涉及浏览器的兼容性和重要性，你现在或以后需要追加不止一个事件给一个元素？如果是，attachEvent和addEventListener是必要的，如果不是，行内事件也是一个妙招。

jQuery和其他js框架包含了不用浏览器执行DOM Level2 事件的方案，所有你能写出跨浏览器的代码不用担心IE的历史

$(ele).on('click', function() {/* do something * /});

不要为了一件事情动用一个框架！

function addEvent(ele, event, funct) {
  if (ele.attachEvent)
  return ele.attachEvent('on'+event, funct);
  else
  return ele.addEventListener(event, funct, false);  
}

学习来源：http://stackoverflow.com/questions/6348494/addeventlistener-vs-onclick

不知道要什么

发表于 2015-05-10   |   分类于 随想

我能想到我小时候被问到长大后想成为什么样的人的回答有：PLA、科学家，后来随着成长肯定有过很多其他想法。虽然幼时的“梦想”有些不切实际，但突然发现，不知从何时起没有了想法，不知道想要什么了。

小时候时间闲，但家里找不到几本书，记得当时从一个抽屉翻出一本金庸的小说《天龙八部》，大冬天的周末晒着太阳看着书挺惬意，年龄太小没看大懂人物关系。囫囵地看了一些没看完，后来那本书被我弄丢了（总之，读过的书很少，记忆中不超过十本，里面有些故事，有机会可以写写）。

进入初中、高中后空闲时间就少了，尤其是高中，目标就“明确”了，家里、长辈、老师念叨的，要好好读书，考上大学。那似乎成为和我一样同龄人当时奋斗的目标。于是学习占据了我青少年期间绝大部分时间。

好不容易考上大学，终于可以松口气了，要填报志愿却发现自己不知该填什么。喜欢什么，不知道；以后想从事什么，也不知道，支援填的稀里糊涂。进了大学一切束缚都解脱了，这下反而不适应了，没有管制了才发现自己缺乏自制力。考上之前，大人们描绘出各种美化、希望，其他一切问题都考上再说。进来后才发现考大学其实只是人生一个小门槛，大学生早已不是天之骄子，满大街的大学生，等待着他们的是紧张高中生活后“自由”的大学生活、毕业时的就业问题、毕业后生活的压力。父辈们用他们的经验和认知去指导子女的人生，却不知时代早已改变，而我们没学会认识自己。

突然发现，从没弄清自己想要什么！从前都在完成长辈的“目标”，没有了目标之后，在需要自己作出选择时却发现自己不清楚自己要什么。习惯于被安排，却从未问过自己，想要什么，最不了解的原来是自己。

有些道理没有人教，那就得自己懂得。人们常在道德上批评贪婪。其实，不知道自己的追求比贪婪更可怜。欲望是人的本性，通过合理途径争取达到自己的目标无可厚非，可笑的时很多时候不知道自己要追求什么。知道自己要什么，也就明确了目标，才会为之努力，那样的辛苦才不会都是苦闷和委屈。

读万卷书，行万里路，人的经历决定他的见识，这也是为什么很多条件不错的家庭从小带子女出去，因为这个过程开拓了孩子的视野，有更多机会让他/她们了解自己。我一直对于《天堂电影院》中老艾费多最后和多多谈话的情景印象深刻。人的一切经历，每次恋爱，每次成功，每次失败，最终都是让你认识自己。当你了解自己，才知道自己要什么，才有可能获得想要的。多和自己聊聊天，去丰富自己，认识自己，不要为了生计，工作忙忙碌碌，浑浑噩噩，日复一日。问问自己要什么。
